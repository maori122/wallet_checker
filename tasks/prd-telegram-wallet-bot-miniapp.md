# PRD: Telegram-бот и Mini App для отслеживания кошельков и ссылок

## Введение / обзор

Нужен **Telegram-бот** и **Telegram Mini App** (Web App), в которых пользователь управляет списком **своих** отслеживаемых криптокошельков, **адресной книгой «знакомых» кошельков** (чужие адреса с подписями — для подписи в уведомлениях) и персональных ссылок, получает **уведомления в чат с ботом**. Ключевое требование — **максимальная анонимность**: изоляция данных между пользователями, минимум персональных данных, желательно шифрование так, чтобы у администратора не было «простого» доступа к содержимому.

**Хостинг (предпочтение):** Cloudflare (Workers + при необходимости D1/KV/Pages). **Резерв:** VPS.

**Важно по Cloudflare:** для Telegram-бота обычно используют **webhook** (POST на Worker) — это **совместимо** с Cloudflare Workers. Mini App — статика на **Pages**, бизнес-логика и API — **Workers**. Периодические проверки блокчейна — **Cron Triggers** Workers (с учётом лимитов времени CPU и квот). **Long polling** на Workers нежелателен; при отказе от webhook понадобится отдельный процесс (часто VPS или контейнер). Если понадобятся тяжёлые собственные индексаторы или постоянные WebSocket-к подключения к нодам — разумнее **VPS** или гибрид (CF + внешние API).

## Цели

1. Пользователь может добавить до **10 кошельков**, просмотреть и удалить свои записи.
2. Пользователь может добавлять **ссылки с произвольными названиями**, редактировать название, удалять, просматривать список.
3. Бот отправляет **уведомления** о событиях: сервисные + по кошелькам — **только входящие переводы не ниже порога**, порог задаётся в **«Настройках»** (зафиксировано заказчиком: **2B**).
4. **Анонимность и безопасность:** данные пользователей не пересекаются; **серверное шифрование at-rest**, ключ вне БД (**4A**); rate limiting от спама.
5. Пользователь может вести **список знакомых адресов с подписями**; при входящем переводе на **свой** отслеживаемый кошелёк, если отправитель совпал с записью в книге, в уведомлении показывается **подпись** (например, «Паша») и сумма (в т.ч. **USDT** и при необходимости **оценка в USD**).
6. **MVP** доставлен на выбранной инфраструктуре (CF или VPS) с простым меню на кнопках.

## Пользовательские истории

1. **Как пользователь**, я хочу добавить кошелёк **Bitcoin или EVM** в отслеживание, чтобы получать уведомления о **входящих** переводах **не ниже заданного порога** (**1C**, **2B**).
2. **Как пользователь**, я хочу видеть до 10 кошельков и удалять лишние, чтобы не превышать лимит и держать список актуальным.
3. **Как пользователь**, я хочу сохранять **личные закладки** (URL + название) без мониторинга сайтов и без напоминаний (**3D:** расширенные сценарии не нужны — только каталог ссылок).
4. **Как пользователь**, я хочу получать сообщения только в своём чате с ботом, чтобы никто другой не видел мои данные.
5. **Как пользователь**, я не хочу вводить email и лишние ПДн — только то, что неизбежно для Telegram (например, `telegram_user_id` для привязки).
6. **Как пользователь**, я хочу сохранить **адрес знакомого** и **короткое имя** (например, «Паша»), чтобы при переводе **на мой** кошелёк уведомление говорило **от кого** пришли средства, а не только «неизвестный адрес».
7. **Как пользователь**, я хочу видеть в уведомлении **сумму и актив** (например, **250 USDT**) и по возможности **примерно в фиате (USD)** для быстрого понимания.

## Функциональные требования

### Общее

1. Система должна идентифицировать пользователя по **Telegram User ID** (из `initData` Mini App и из апдейтов бота), без публичного отображения ID другим пользователям.
2. Интерфейс бота: **кнопки** (reply/inline keyboard), без обязательности запоминать команды; пункты меню: **«Мои кошельки»**, **«Знакомые кошельки»** (или эквивалентное короткое название в EN), **«Мои ссылки»**, **«Настройки»**. Тексты интерфейса — **русский и английский** с **переключателем языка в «Настройках»** (**5C**).
3. Mini App должен позволять выполнять те же основные операции, что и бот (согласованность данных через один backend API).

### Кошельки

4. Пользователь может **добавить** кошелёк типа **Bitcoin** или **EVM** (в MVP — минимум **Bitcoin mainnet** и **одна EVM-сеть**, по умолчанию **Ethereum mainnet**; список EVM-сетей можно расширить во второй итерации без смены модели данных) с валидацией формата адреса под выбранный тип.
5. Пользователь может **удалить** кошелёк из своего списка.
6. Пользователь может **просмотреть** список только своих кошельков.
7. Система не позволяет хранить более **10** активных кошельков на одного пользователя (отказ с понятным сообщением).

### Знакомые кошельки (адресная книга контрагентов)

*Отдельно от «моих» отслеживаемых кошельков: это **чужие** адреса, которые пользователь сам помечает именами, чтобы узнавать отправителя во входящих уведомлениях.*

8. Пользователь может **добавить запись**: **адрес** (BTC или EVM в рамках поддерживаемых сетей MVP) + **подпись** (отображаемое имя, например «Паша»), длина подписи с разумным лимитом (например **до 40 символов**).
9. Пользователь может **изменить подпись** и **удалить** запись.
10. Пользователь может **просмотреть** список только своих знакомых кошельков.
11. Система не позволяет хранить более **50** записей в адресной книге на пользователя (лимит настраиваемый константой; цель — защита от спама и перегруза сопоставлений).
12. **Сопоставление при уведомлении:** для входящего перевода на **отслеживаемый** адрес пользователя определяется **адрес отправителя** (`from` / аналог в модели индексатора). Если он **точно совпадает** (с учётом нормализации: checksum **0x** для EVM, формат BTC) с адресом из адресной книги этого пользователя — в текст уведомления подставляется **подпись**. Если совпадения нет — в шаблоне используется **нейтральная формулировка** (например, «Неизвестный отправитель» + сокращённый адрес), без утечки данных других пользователей.
13. Запрещено показывать одному пользователю подписи или адреса из книги **другого** пользователя.

### Ссылки

14. Пользователь может **добавить** ссылку (URL) и **название**; **без** мониторинга содержимого страниц и **без** напоминаний по расписанию — только хранение и открытие (**3D**).
15. Пользователь может **изменить название** существующей ссылки.
16. Пользователь может **удалить** ссылку.
17. Пользователь может **просмотреть** список только своих ссылок.

### Уведомления

18. Бот отправляет уведомления пользователю **в личный чат** с ботом (тот же Telegram User ID).
19. **Сервисные уведомления:** подтверждения мутаций (кошелёк / знакомый / ссылка), тестовое сообщение из «Настроек» (опционально).
20. **По «моим» кошелькам:** уведомление только при **входящей** транзакции на отслеживаемый адрес, если сумма **не ниже порога** (**2B**). Для **BTC** — порог в нативной единице (как зафиксировано ранее). Для **EVM** в MVP дополнительно поддерживаются **входящие ERC-20**, минимум **USDT** на **Ethereum mainnet** (сценарий «Паша перевёл USDT Саше»): порог для **USDT** задаётся **в USDT (≈ USD)** отдельным полем в «Настройках»; для **native ETH** — отдельный порог в ETH (как ранее). *Итого в настройках пороговые поля: BTC, ETH, USDT (или объединённая UX-форма «пороги по активам»).*
21. **Текст уведомления о входящем:** содержит **на какой мой адрес** пришло (маскирование/сокращение допустимо), **актив и сумму**, при совпадении отправителя с адресной книгой — **подпись отправителя** (п. 12), и при включённой настройке — **оценку в USD** для отображаемого актива (курс с внешнего **price** API, округление и дисклеймер «≈»).
22. Дедупликация: одно и то же событие цепочки не должно приводить к лавине повторных сообщений (идемпотентность по `txid` + типу события / логу индексатора).

### Безопасность и анонимность

23. API и хранилище должны гарантировать, что запросы выдают/меняют данные **только** владельца (авторизация по подписанному `initData` в Mini App и по `telegram_user_id` из доверенного контекста бота).
24. В логах не хранить **секреты** и по возможности не хранить полные URL/адреса в открытом виде (маскирование в логах).
25. **Шифрование at-rest (4A):** поля с **моими** кошельками, **знакомыми** адресами и подписями, ссылками и названиями хранятся в БД в зашифрованном виде; мастер-ключ — в **Secrets** окружения; производная **per-user** соль/ключ от `user_id` (детализация в тех. дизайне). Цель — отсутствие чтения «в лоб» из дампа БД без ключа.
26. **Rate limiting** на мутации и на дорогие операции (лимиты в минуту/час на пользователя), ответ HTTP 429 или аналог в Telegram flow.

### Настройки

27. Раздел **«Настройки»**: **переключатель языка RU/EN** (**5C**); **пороги уведомлений:** BTC, ETH (native), USDT (ERC-20); переключатель **«показывать оценку в USD (≈)»** в тексте входящих; включение/выключение **блокчейн-уведомлений** и/или сервисных (минимум master-switch для пушей из бота); тестовое уведомление; краткая справка (без сбора email).

## Не-цели (вне рамок MVP)

- Мониторинг изменений веб-страниц по сохранённым ссылкам, напоминания по расписанию для ссылок.
- «Верификация личности» отправителя: подпись в адресной книге — **только метка пользователя**, система не гарантирует, что владелец ключа — именно этот человек.
- Социальные функции, шаринг списков между пользователями.
- KYC, email-регистрация, привязка банковских карт.
- Собственный полноценный блокчейн-индексатор «с нуля».
- Гарантия «нулевого знания» для администратора без участия пользователя (E2E с паролем на клиенте) — **не входит в MVP**; зафиксировано **4A:** серверное шифрование, ключ в Secrets.

## Дизайн / UX (опционально)

- Простые экраны Mini App: разделы **мои кошельки**, **знакомые**, **ссылки**, **настройки** + формы добавления/редактирования; строки UI — **RU/EN** по сохранённой настройке.
- В боте — кнопочное меню, подтверждение удаления (да/нет); сообщения бота — на выбранном языке.

## Технические соображения

### Вариант A: Cloudflare

- **Worker** — webhook Telegram, REST API для Mini App, проверка подписи `initData`.
- **D1** (или KV для очень простых структур) — метаданные пользователя, зашифрованные полезные нагрузки.
- **Pages** — сборка Mini App (React/Vanilla).
- **Secrets** — `TELEGRAM_BOT_TOKEN`, `TELEGRAM_WEBHOOK_SECRET`, мастер-ключ шифрования.
- **Cron** — опрос внешних API блокчейна по сохранённым адресам (после выбора провайдеров и сетей).

### Вариант B: VPS

- Контейнер или Node/Bun + PostgreSQL/SQLite + reverse proxy (HTTPS).
- Те же принципы: webhook, шифрование, rate limit (например, Redis или in-memory + персистентные лимиты в БД).

### Зависимости

- Telegram Bot API.
- Для уведомлений по кошелькам — внешние **indexer/RPC** для **BTC**, **ETH** и **логов ERC-20 (USDT)** на **Ethereum mainnet**; для строки «≈ $» — **курсы** (price API) с кэшированием и лимитами запросов.

## Метрики успеха

- Успешное добавление/удаление **своего** кошелька, записи **знакомого** и ссылки в боте и в Mini App с согласованным состоянием.
- Невозможность получить чужие списки при подмене `user_id` в запросе (негативные тесты).
- Rate limit срабатывает при искусственном спаме (smoke-тест).
- Данные в БД не читаются как plain text без ключа (проверка выборочной записи).
- При тестовом входящем переводе **ниже порога** уведомление **не** отправляется; при **≥ порога** — одно сообщение на событие (дедупликация).
- Если `from` совпадает с записью в **знакомых**, в тексте уведомления отображается **подпись**; при несовпадении — шаблон для неизвестного отправителя.

## Зафиксированные решения заказчика

| № | Тема | Выбор |
|---|------|--------|
| 1 | Сети MVP | **C:** **Bitcoin + EVM** (в PRD зафиксировано: BTC mainnet + Ethereum mainnet как базовая EVM-сеть). |
| 2 | Уведомления по кошелькам | **B:** только **входящие** переводы **не ниже порога**; порог в **«Настройках»**. |
| 3 | Ссылки | **D + «не нужны»** трактовано как: **не** нужны мониторинг и напоминания — только **закладки** (CRUD + открыть). *Если имелось в виду полное исключение раздела «Мои ссылки» из продукта — сообщите, скорректируем объём MVP.* |
| 4 | Конфиденциальность от админа | **A:** серверное **at-rest** шифрование, ключ в **Secrets** (не E2E с паролем пользователя). |
| 5 | Язык UI | **C:** **RU + EN**, переключатель в **«Настройках»**. |
| 6 | Знакомые кошельки | **Доп. фича (заказчик):** адресная книга **чужих** адресов с **подписью**; во входящем уведомлении на **мой** кошелёк — показ **от кого** (если адрес отправителя в книге), пример: **«Паша перевёл вам 250 USDT (≈ $250)»** при включённой оценке USD. |

### Оставшиеся вопросы (техдизайн, не блокируют старт)

- Расширение **ERC-20** сверх **USDT** (USDC и т.д.) — по желанию после MVP.
- Уточнить единицы порога в UI: **BTC** (сатоши/mBTC), **ETH** (wei/ETH), **USDT** (с дробной частью) с валидацией и подсказками.
- Граничные случаи **EVM**: вход через контракт / **внутренние** переводы бирж — когда `from` не равен «кошельку знакомого», показывать неизвестного отправителя (ожидаемое поведение; в справке можно кратко объяснить).

## Приложение: ответ на вопрос «возможен ли Cloudflare?»

**Да, для типичного MVP — возможен:** webhook бота, Mini App на Pages, данные в D1/KV, секреты в Cloudflare, периодические задачи через Cron и внешние blockchain API. **Переход на VPS** имеет смысл при long polling, тяжёлой очереди задач, долгих воркеров или нехватки лимитов Workers под выбранные сети и объём опросов.
